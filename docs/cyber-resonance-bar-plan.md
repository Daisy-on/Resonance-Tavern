# 共振酒吧（Cyber Resonance Bar）项目计划书

> 核心理念：玩家不是在“背配方”，而是在用酒液、节奏与波形去弥合客人的情绪频率。

## 1. 项目概述

### 1.1 项目定位

- 类型：赛博朋克像素风、酒馆模拟经营、精准操作调酒、波形匹配、文字叙事
- 参考气质：`VA-11 Hall-A` 的酒吧叙事氛围，`红弦俱乐部` 的情绪调酒表达，叠加更强的操作性与经营压力
- 平台：PC / 移动端兼容的纯前端 H5 单机游戏
- 技术限制：优先控制在 8MB 以内，无强依赖网络，无后端服务
- 核心卖点：
  - 调酒动作直接塑造“情绪波形”
  - 经营资源形成持续压力与成长动力
  - 客人与故事提供长期留存
  - 共振码带来轻量离线社交与“濒死翻盘”体验

### 1.2 目标用户

- 喜欢 `VA-11 Hall-A`、`红弦俱乐部`、像素叙事游戏的玩家
- 喜欢轻中度经营、数值成长、反复挑战的玩家
- 希望在碎片时间获得“沉浸 + 放松 + 微刺激”的上班族与学生群体

### 1.3 一句话体验目标

玩家在霓虹闪烁的像素酒吧里，通过精准又自由的调酒动作绘制情绪波形，安抚一个个赛博都市中的灵魂，同时在资金、电力、评分三重压力下尽可能活得更久。

## 2. 设计支柱

### 2.1 支柱一：调酒必须“有手感”

- 玩家每个动作都应立刻改变液体状态、波形形态、音效反馈
- 不做“菜单式配方点击”，而做“动作驱动的实时反馈”
- 新玩家能凭直觉上手，熟练玩家能靠理解映射规则追求高分

### 2.2 支柱二：调酒不是解题，而是共鸣

- 客人需求不只是一条曲线，也是一句情绪描述
- 玩家看到的是“一个人现在需要怎样的酒”，不是“公式题”
- 完美匹配不仅给钱，还推进人与人之间的关系

### 2.3 支柱三：经营压力要持续存在

- 每单都影响资金、电力、评分
- 任何一项归零即失败，迫使玩家权衡风险
- 成长的本质不是堆数值，而是扩展解决复杂需求的能力

### 2.4 支柱四：每一局都要有新鲜感

- 无限天数挑战
- 随机事件改变当日策略
- 客人组合、订单波形、隐性顺序要求、特殊剧情共同形成重玩价值

## 3. 核心循环

### 3.1 单局外循环

1. 进入新的一天
2. 结算租金、设备耗电、随机事件
3. 接待客人并完成调酒订单
4. 获得报酬、好感、资源变化、剧情推进
5. 用奖励购买酒类、工具、设备升级
6. 一天结束后自动存档并进入下一天
7. 任一核心资源归零则本局结束

### 3.2 单单内循环

1. 客人入座并给出目标波形与情绪描述
2. 玩家选择基酒、辅料、工具并执行操作
3. 系统实时更新当前酒液属性与蓝色波形
4. 玩家观察与目标红色波形的匹配度变化
5. 选择继续微调或直接“上酒”
6. 系统根据匹配度、顺序要求、特殊事件结算收益与后果

## 4. 核心玩法：波形调酒系统

### 4.1 玩法目标

玩家需要通过调酒动作精准操控一组酒液属性，让酒液对应的预览的蓝色波形尽可能逼近客人的目标红色波形。

### 4.2 调酒台布局

- 配料架：基酒、利口酒、糖浆、果汁、苏打水、柠檬、薄荷叶、苦精、特殊素材
- 工具区：量酒器、调酒勺、摇酒壶、捣棒、冰块桶、榨汁器、喷枪
- 酒杯区：显示当前液体层次、颜色、气泡、容量
- 示波器区：左侧显示目标波形，右侧或叠层显示当前波形
- 提交区：`上酒`、`倒掉重做`、剩余容量、匹配度、特殊提示

### 4.3 六项隐藏属性

每杯酒维护以下核心属性，范围推荐为 `0-100`：

- 烈度 `strength`
- 甜度 `sweetness`
- 酸度 `acidity`
- 温度 `temperature`
- 气泡感 `sparkle`
- 混合度 `blend`

后续扩展属性：

- 烟熏度 `smoke`
- 芳香度 `aroma`
- 稀释度 `dilution`
- 氧化度 `oxidation`

### 4.4 操作与属性映射

| 操作 | 主要影响 | 次要影响 | 波形反馈 |
| --- | --- | --- | --- |
| 选择基酒 | 烈度、基调 | 芳香度 | 决定基础波形类型 |
| 倒酒（控制毫升） | 烈度 | 容量占用 | 振幅提升 |
| 加糖浆 / 果汁 | 甜度、酸度 | 颜色变化 | 频率变化、对称性变化 |
| 加冰块 | 温度下降、稀释度上升 | 平滑度 | 波形衰减更快、振幅变缓 |
| 摇荡 | 混合度上升 | 氧化度上升 | 波形更平滑，过量则出现不和谐高频 |
| 搅拌 | 混合度小幅上升 | 芳香度稳定 | 相位更稳定，细节更可控 |
| 捣碎 | 酸度、芳香度上升 | 不规则度上升 | 局部波形扭曲 |
| 加苏打水 | 气泡感上升 | 酸度微升 | 增加高频噪声与毛刺 |
| 喷枪 / 点燃 | 温度上升 | 烟熏度上升 | 波形出现瞬态尖峰 |

### 4.5 基酒与基础波形建议

| 基酒 | 风味倾向 | 建议基础波形 | 玩法定位 |
| --- | --- | --- | --- |
| 伏特加 | 冷、直、强烈 | 正弦波 | 通用型、高振幅基础 |
| 琴酒 | 清、草本、锐利 | 三角波 | 中频细调能力强 |
| 朗姆 | 甜、厚、热带感 | 平滑锯齿 | 适合甜度驱动组合 |
| 威士忌 | 沉、木质、厚重 | 方波 | 低频、强存在感 |
| 白兰地 | 甜、厚、热带感 | 阻尼波 | 适合甜度驱动组合 |
| 龙舌兰 | 烈、刺、边缘感 | 破碎脉冲 | 做特殊订单与高风险波形 |

说明：基础波形不是硬编码配方答案，而是玩家理解风味个性的视觉化入口。

### 4.6 波形参数映射建议

| 属性 | 波形参数 | 公式建议 |
| --- | --- | --- |
| 烈度 | 振幅 `amplitude` | `0.4 + strength / 80` |
| 甜度 | 频率 `frequency` | `1.0 + sweetness / 35` |
| 酸度 | 左右偏斜 / 对称性 | `symmetry = 1 - acidity / 140` |
| 温度 | 衰减速度 | `decay = 0.1 + (100 - temperature) / 150` |
| 气泡感 | 高频噪声 | `noise = sparkle / 120` |
| 混合度 | 平滑度 | `smooth = blend / 100` |
| 烟熏度 | 尖峰触发率 | `spikeChance = smoke / 180` |

### 4.7 匹配算法建议

- 波形采样点数：`128`
- 实时计算：每次操作后增量更新
- MVP 匹配方式：归一化后逐点均方误差 `MSE`
- 进阶匹配方式：在高难度客人或动态目标中引入 `DTW`
- 最终匹配度：

```text
score = clamp(100 - mse * 120, 0, 100)
```

- 附加修正项：
  - 操作顺序命中：`+5 ~ +15`
  - 溢杯：`-10 ~ -25`
  - 过度稀释：`-5 ~ -20`
  - 随机事件成功适配：额外奖励或惩罚

### 4.8 匹配度分档

| 匹配度 | 评级 | 结果 |
| --- | --- | --- |
| 95-100 | 完美共振 | 高额报酬、大量好感、可能触发特殊剧情 |
| 80-94 | 很满意 | 正常报酬、稳定好感提升 |
| 60-79 | 勉强接受 | 低额报酬、小幅好感变化 |
| 40-59 | 失望 | 少量或拒付，评分下降 |
| 0-39 | 彻底失败 | 拒付、评分重创、电力或资金惩罚 |

### 4.9 容错与深度设计

- 无固定标准配方，允许自由尝试
- 操作顺序可构成“隐藏条件”
- 倒酒过量触发溢出
- 冰块过多导致波形被抹平
- 摇荡过久导致氧化噪音增加
- 部分客人存在“隐性偏好”，例如更喜欢后调型、分层型、冷饮型

## 5. 经营系统

### 5.1 三维核心资源

| 资源 | 获取方式 | 消耗方式 | 归零后果 |
| --- | --- | --- | --- |
| 资金 | 订单收入、小费、事件奖励 | 房租、进货、升级 | 破产 |
| 电力 | 每日补给、事件补充、完美奖励 | 设备运转、故障、失败惩罚 | 停业 |
| 评分 | 高匹配、VIP 好评、剧情推进 | 失败订单、超时、事故 | 客流崩塌 |

### 5.2 日结规则建议

- 每天开始扣除房租、材料费用
- 每单消耗基础电力
- 每天营业结束后按设备等级与天气追加耗电
- 客人连续满意时触发口碑连击，评分提升倍率增加
- 若当天出现连续失败，后续客人的耐心与容忍度下降

### 5.3 失败定义

以下任意一项归零，本局结束：

- 资金 `<= 0`
- 电力 `<= 0`
- 评分 `<= 0`

### 5.4 Game Over 表现建议

- 画面：酒吧卷帘门落下，霓虹灯逐段熄灭
- 文案：`在这个城市，至少你曾听见过某些灵魂的频率。`
- 结算：展示生存天数、总收入、完美调酒次数、解锁客人档案数、最常使用的酒谱风格
- 引导：鼓励玩家用更成熟的经营理解开启下一局

## 6. 成长与商店系统

### 6.1 可购买内容

- 更多酒类：解锁新的基础波形个性
- 更多辅料：增加频率、噪声、相位等操控手段
- 更多工具：开放更复杂操作
- 新设备：提升容量上限、降低误差、减少耗电
- 装饰升级：提高基础评分获取速度，解锁高阶客人

### 6.2 升级层次

| 类型 | 例子 | 功能价值 |
| --- | --- | --- |
| 原料扩容 | 大容量伏特加、双倍冰桶 | 提高容错与营业稳定性 |
| 工具升级 | 精准量酒器、静音摇壶 | 降低输入误差 |
| 设备升级 | 新示波器、节能吧台 | 显示更多辅助信息，减少耗电 |
| 服务升级 | 灯牌、音响、座位改造 | 提高评分与高级客人刷新率 |

### 6.3 成长节奏原则

- 前 3 天让玩家感受到明显成长
- 第 4-7 天引入复杂需求和策略分化
- 7 天后进入无限模式，成长放缓但系统组合增多

## 7. 无限挑战与难度曲线

### 7.1 难度提升方式

- 目标波形更复杂
- 客人描述更抽象
- 顺序要求更隐蔽
- 随机事件更多变
- 原材料成本波动增强
- 高级客人对错误惩罚更重

### 7.2 难度分层

| 阶段 | 天数 | 主要变化 |
| --- | --- | --- |
| 新手期 | 1-3 | 单峰波形、提示明显、容错高 |
| 熟悉期 | 4-7 | 多参数组合、顺序开始重要 |
| 压力期 | 8-14 | 动态目标、资源压力增大 |
| 无限期 | 15+ | 复杂谐波、事件叠加、极限经营 |

### 7.3 复杂订单类型

- 双峰型：需要两段情绪起伏
- 毛刺型：需要苏打与高频成分
- 衰减型：强调冰块与降温控制
- 相位偏移型：强调搅拌与顺序
- 动态平移型：目标波形会缓慢滑动，要求更快调整

## 8. 随机事件系统

### 8.1 设计目标

- 打破固定策略
- 让玩家重新理解已有工具
- 让每一天都存在“今天要换一种活法”的感觉

### 8.2 推荐事件池

| 事件 | 效果 | 玩法影响 |
| --- | --- | --- |
| 雨天 | 客人更偏向低温、平滑型波形 | 冰块与摇荡价值上升 |
| 禁酒令巡查 | 某类烈酒当日受限 | 强迫寻找替代解法 |
| 市场波动 | 特定原料涨价或缺货 | 经营策略改变 |
| 冰块短缺 | 冰块供应减半 | 容错下降 |
| 摇酒壶故障 | 摇荡结果产生随机偏移 | 逼迫改用搅拌与更稳操作 |
| 酸柠檬日 | 酸味操作效果翻倍 | 需要重新估算参数 |
| 网红直播 | 完美订单小费 3 倍，失败评分双倍下跌 | 高风险高收益 |
| 神秘酒谱 | 某客人提供模糊提示 | 成功可解锁隐藏波形或稀有材料 |

### 8.3 事件规则建议

- 每天最多 1 个强事件 + 1 个弱环境事件
- 事件在开始营业前给出预告
- 少数事件可通过对话提前获知，形成叙事与经营联动

## 9. 叙事与角色关系系统

### 9.1 叙事目标

- 让调酒成功不仅是“赚钱”，也是“打开一个人”
- 让角色成为长期留存动力，而不是一次性剧情摆设

### 9.2 NPC 结构模板

每位 NPC 建议包含：

- 基础身份：职业、派系、危险度、消费能力
- 情绪母题：孤独、愤怒、麻木、理想主义、背叛、救赎
- 波形偏好：偏好的基础波形与操作方式
- 好感阈值：`初识 / 熟悉 / 信任 / 共鸣`
- 专属奖励：剧情、工具、酒材、主题曲、头像

### 9.3 好感获取逻辑

- 成功订单：`+1`
- 高匹配：`+2`
- 完美共振：`+4`
- 连续服务成功：额外奖励
- 对话选择命中角色偏好：小幅增加
- 连续失败或严重失配：减少好感或冻结剧情

### 9.4 档案系统

- 每个角色拥有若干“记忆碎片”
- 解锁条件来自：
  - 高匹配订单次数
  - 特定随机事件中成功服务
  - 特定对话路径
  - 特殊波形订单完成
- 集齐后点亮档案：
  - 霓虹头像
  - 专属背景音乐变奏
  - 人物总结文本
  - 稀有奖励或新事件入口

### 9.5 对话节奏建议

- 普通客人：单局 2-4 句短对话
- 核心角色：按多日逐步展开
- 深度剧情只在高好感或关键节点展开
- 不让对话过长，避免破坏经营节奏

## 10. 离线社交：共振码系统

### 10.1 设计目标

- 在纯单机环境下实现“玩家互相救一把”
- 激发分享欲和讨论欲
- 保持实现成本低、使用门槛低

### 10.2 共振码来源

将玩家最近 3 次完美调酒的关键数据编码为短字符串：

- 基酒编号
- 关键操作序列
- 每步参数区间
- 当时的评语种子

### 10.3 使用场景

#### 濒临破产救援

- 当玩家某项资源即将见底时可输入好友共振码
- 触发“好友幻影客人”来店
- 幻影点一杯与该码绑定的酒
- 系统给玩家有限提示，让玩家复现操作
- 成功则获得大额奖励、缓解破产压力

#### 日常 VIP 来访

- 非破产状态也可录入共振码
- 该幻影作为常驻 VIP 偶尔刷新
- 点固定风格订单，提供小额奖励与社交彩蛋

### 10.4 评语与分享欲设计

系统根据波形特征自动生成“玄学短评”，用于分享界面：

- 高频高振幅：`像在坠落的高楼边缘起舞`
- 低频高平滑：`像大雨里一盏愿意等你的灯`
- 高毛刺高酸度：`像把心事咬碎后咽下去`

目标是让玩家愿意截图传播，而不是只分享一串冷冰冰的编码。

## 11. 视听风格方案

### 11.1 美术方向

- 赛博朋克像素风
- 色彩关键词：青绿霓虹、深红警示、旧金属灰、雨夜蓝紫
- 画面感觉：压抑都市 + 温暖角落

### 11.2 画面重点

- 吧台与器具采用粗像素轮廓
- 杯中液体使用半透明渐变和分层效果
- 波形采用霓虹描边与轻微拖尾
- 完美共振时触发 `Cyber Zen` 视觉奖励：
  - 液体出现像素尘埃
  - 杯中脉冲轻闪
  - 波形边缘短暂辉光增强

### 11.3 关键 UI 原则

- 重要信息永远可一眼看到：资源、目标波形、当前匹配度、容量
- 操作区尽量少层级、少弹窗
- 色彩反馈必须服务玩法，而不是只做装饰

### 11.4 音频方向

- 使用 Web Audio API 合成背景音乐与操作音效
- 操作音效与行为绑定：
  - 倒酒：短流体噪声
  - 冰块：清脆点击
  - 摇荡：节奏颗粒噪声
  - 完美命中：和声音程收束
- 背景音乐与匹配度联动：
  - 匹配越高，和声越稳定
  - 失败越多，底噪与失真略增

## 12. 技术实现方案

### 12.1 技术选型

- 渲染：`HTML5 Canvas`
- 音频：`Web Audio API`
- 状态管理：原生 TypeScript 模块或轻量状态机
- 存档：`localStorage`
- 打包：`Vite`

### 12.2 核心模块拆分建议

### 12.4 性能与体积控制

- 不依赖大型引擎
- 不使用体积重的图片与音频资源
- UI、美术、液体、图标尽量代码绘制
- 剧情文本做分段压缩与按需加载
- 波形采样维持在 `128` 或 `256` 以内
- 移动端优先锁定 `30-60 FPS`

### 12.5 存档策略

- 每单结束自动存档
- 每天结算后强制存档
- 保留 1 份当前档 + 1 份应急回滚档
- 存档内容：
  - 天数
  - 资源
  - 已购道具
  - 客人好感
  - 已解锁剧情
  - 当前事件池权重
  - 最近完美调酒记录

- 经营压力是否足够驱动继续游玩
- 客人对话是否能成为正反馈，而非打断节奏

### 15.2 内容制作优先级

3. 3 位足够有辨识度的核心 NPC
4. 基础事件池
5. 共振码与分享文案

## 16. 风险与应对

| 风险 | 描述 | 应对 |
| --- | --- | --- |
| 手感不直观 | 玩家看不懂波形变化 | 增加操作提示、渐进解锁、训练模式 |
| 数值过于复杂 | 新玩家被系统劝退 | MVP 只保留 6 项核心属性 |
| 叙事打断节奏 | 对话过长影响经营快感 | 对话短句化，深剧情放在高好感节点 |
| 体积超限 | 像素资源与文本过多 | 坚持代码绘制与文本压缩 |
| 后期重复 | 无限模式失去新鲜感 | 强化事件池、隐藏顺序需求、VIP 客人 |

## 17. 成功标准

### 17.1 原型阶段成功标准

- 玩家在 3 单内理解基本波形操作
- 至少愿意主动重试 1 次失败订单
- 能清楚感知不同基酒与工具的差异

### 17.2 可玩版成功标准

- 单局游玩时长达到 15-30 分钟
- 玩家会主动尝试解锁新客人或更高天数
- 共振码具备被截图分享的冲动点

## 18. 结语

这不是一款单纯关于“调酒”的游戏，而是一款关于“如何在崩坏城市里，精确地回应他人情绪”的游戏。

玩家在这里经营的不只是酒吧，也是一个赛博时代的情绪避难所。每一次倒酒、每一次摇荡、每一次波形接近，都是一次人与人之间短暂却真实的同频。

如果项目执行遵循 `先手感、后内容；先原型、后扩张；先系统闭环、后世界堆叠` 的原则，这个题材完全有机会做出鲜明辨识度和持续游玩价值。

## 19. 执行优化：从方案到可开发版本

这一章的目标不是继续扩写创意，而是把前文压缩成一个真正能开工、能验收、能迭代的执行路径。

### 19.1 当前文档再评估

当前方案的优点很明显：

- 主题辨识度强，`调酒 + 波形 + 情绪共鸣` 的核心概念成立
- 核心循环清楚，已经具备 `接单 -> 操作 -> 结算 -> 成长 -> 下一天` 的闭环雏形
- 系统层次完整，包含经营、角色、事件、成长、离线社交等长期内容方向
- 技术方向务实，`Canvas + Web Audio + localStorage + Vite` 足够支撑目标

当前方案最主要的问题也很明确：

- 创意层足够丰富，但“第一版到底砍到哪里”还不够硬
- 波形玩法定义得多，但玩家输入、反馈、容错边界还不够具体
- 经营系统有资源框架，但缺少可直接录入代码的首轮数值锚点
- 叙事、事件、商店、共振码都想做，容易在原型期分散精力
- 缺少模块级交付定义，开发时容易陷入“系统都做了一点，但没有一个完整可玩”

结论：

这份文档已经足够做“立项与方向判断”，但还不足以直接支撑高效率开发。接下来必须把它收束成 `一个明确的 MVP 闭环 + 一套阶段验收标准 + 一张暂缓清单`。

### 19.2 第一原则：先做出一个能连续玩 10 分钟的闭环

原型阶段不要追求“大而全”，只追求下面这件事成立：

- 玩家能看懂订单需求
- 玩家能通过 3-5 个操作明显改变波形
- 玩家能感到“调得更准就赚得更多”
- 玩家会在资源压力下做下一单，而不是玩完一单就退出

如果这四点没有成立，那么：

- 不要先做更多客人
- 不要先做更深剧情
- 不要先做共振码
- 不要先做复杂音频联动

### 19.3 建议重定义 MVP 边界

建议把现在的 MVP 再压缩一层，定义为：

- 仅做 `1 个营业场景`
- 仅做 `3 种基酒`：伏特加、琴酒、威士忌
- 仅做 `5 个操作`：倒酒、加糖浆、加冰、搅拌、加苏打水
- 仅做 `3 个核心属性` 参与匹配：烈度、甜度、温度
- 保留 `气泡感` 作为辅属性，只影响视觉和少量分数修正
- 暂时移除 `混合度、烟熏度、氧化度、芳香度` 的正式数值化
- 仅做 `3 位基础客人`
- 仅做 `3 类订单模板`
- 仅做 `资金 / 电力 / 评分` 三资源
- 仅做 `1 个商店页 + 6 个可购买升级`
- 仅做 `3 个随机事件`

这样做的目的不是削弱特色，而是先保证最重要的辨识度真的成立。

### 19.4 首个可玩版本的最小闭环定义

首个“可玩版本”应满足以下流程：

1. 进入新的一天
2. 显示今日房租、基础电力、今日事件
3. 连续接待 `5` 位客人
4. 每位客人给出一句情绪描述 + 一张目标波形卡
5. 玩家在限定容量内完成调酒并提交
6. 系统根据匹配度结算金钱、电力、评分、好感
7. 第 `5` 单结束后进入日结
8. 玩家进入商店购买 1 次升级
9. 自动存档并进入下一天
10. 任一资源归零则 Game Over

这里要注意两个边界：

- 单日固定 `5` 单，优先保证节奏清晰，不要一开始就做无限客流
- 商店固定每天只购买 `1` 次，减少数值爆炸和 UI 复杂度

### 19.5 单单流程状态机

调酒系统要先写成明确状态机，而不是散落在多个脚本里的 if 判断。

推荐状态如下：

```ts
type OrderFlowState =
  | "guest_enter"
  | "dialogue"
  | "order_preview"
  | "mixing"
  | "submit_confirm"
  | "result"
  | "resource_settlement"
  | "relationship_update"
  | "order_end";
```

每个状态必须有 3 个定义：

- `进入条件`
- `允许输入`
- `退出条件`

例如：

- `order_preview`：只允许查看目标波形、文字描述、特殊提示
- `mixing`：只允许执行操作、撤销最后一步、提交、倒掉重做
- `result`：只展示匹配度、评价、资源变化，不允许继续改酒

这一步会极大降低后面接入 UI、音效、教程时的混乱度。

### 19.6 MVP 输入与交互定义

如果目标是 `PC / 移动端兼容`，那原型阶段就不能依赖复杂手势。

建议 MVP 输入如下：

- 点击基酒按钮：选择基酒
- 长按倒酒按钮：按时间增加毫升
- 点击糖浆按钮：固定增加一次甜度
- 点击冰块按钮：固定加入 `1` 枚冰块
- 点击搅拌按钮：执行一次搅拌
- 点击苏打按钮：固定增加一次气泡感
- 点击 `上酒`：提交
- 点击 `倒掉重做`：清空当前杯子

原型阶段先避免以下高成本输入：

- 真实拖拽倒酒轨迹
- 摇晃手机体感
- 多点触控混合操作
- 复杂分层拉花类交互

先把“参数变化可理解”做出来，比“动作表演感很强”更重要。

### 19.7 首轮数值锚点建议

为了避免系统空转，建议先给一组可直接写死测试的初始值：

#### 局内初始资源

- 初始资金：`120`
- 初始电力：`18`
- 初始评分：`40`
- 每天固定房租：`30`
- 每日基础补电：`12`

#### 单单基础消耗

- 每开一单固定消耗电力：`1`
- 每执行一次操作额外消耗电力：`0.2`
- 倒掉重做额外扣钱：`3`
- 超过容量直接判定溢杯：评分 `-3`

#### 订单收益建议

```text
baseReward = 18 + difficulty * 6
scoreBonus = floor(baseReward * (matchScore - 60) / 40)    // 仅当 matchScore >= 60 生效
tip = matchScore >= 90 ? 6 + difficulty * 2 : 0
finalReward = max(6, baseReward + scoreBonus + tip)
```

#### 评分变化建议

- `95-100`：评分 `+4`
- `80-94`：评分 `+2`
- `60-79`：评分 `0`
- `40-59`：评分 `-3`
- `0-39`：评分 `-6`

这组数值的意义是：

- 好单明显赚钱
- 普通单能勉强维持
- 差单会让经营压力真实存在
- 玩家必须追求稳定，而不是只靠一次大单翻盘

### 19.8 订单生成不要一开始就全随机

原型期最容易失控的点，是“波形全程序生成，但不知道难度是否合理”。

建议首版订单采用 `模板 + 微随机`：

- 先手工做 `9` 个订单模板
- 每个模板绑定一种情绪描述、一种基础波形、一组难度标签
- 允许系统只在 `振幅 / 频率 / 衰减` 上做小范围扰动
- 客人的文本与波形模板要成对设计，不要完全脱钩

推荐首批订单类型：

- 平稳安抚型：低频、低温、低噪声
- 直烈提神型：高振幅、中低频
- 轻快跳跃型：中振幅、高频、少量气泡

这样可以先验证“玩家是否真的理解映射规则”，而不是把问题藏进随机性里。

### 19.9 首批内容最小数据集

如果要开始正式建表，建议先只准备下面这些数据：

#### 酒类数据

- `3` 种基酒
- `2` 种甜味辅料
- `1` 种气泡辅料
- `1` 种冰块配置

#### 客人数据

- `3` 位常驻客人
- 每人 `4` 段初始对话
- 每人 `3` 个订单池权重
- 每人 `2` 个好感节点

#### 事件数据

- `雨天`
- `冰块短缺`
- `网红直播`

#### 商店数据

- 容量 +10%
- 初始电力 +2
- 日租 -5
- 高评分额外小费概率 +10%
- 倒酒误差容忍 +1 档
- 事件预告信息更明确

只要这些数据能跑起来，游戏已经具备“可反复测试”的骨架。

### 19.11 模块开发顺序建议

建议严格按下面顺序实现，不要跳着做：

1. `drink-state` 与 `mix-actions`
2. `wave-generator`
3. `wave-match`
4. `order-flow state machine`
5. `economy-system`
6. `1 位客人 + 1 天流程`
7. `商店`
8. `存档`
9. `事件`
10. `第 2-3 位客人`

原因很简单：

- 前 4 步决定“玩法本身是否成立”
- 第 5-8 步决定“能不能形成经营闭环”
- 第 9-10 步才是“是否足够丰富”

### 19.12 六周开发路线图

相比“按功能模块想”，更推荐按“可交付版本”推进。

#### 第 1 周：做出可操作酒杯

目标：

- 进入主界面
- 能选择基酒
- 能执行 5 个基础操作
- 杯中属性实时变化
- 波形实时更新

验收标准：

- 玩家不看代码也能理解每个按钮的直接效果
- 至少 `3` 个操作会让波形肉眼明显变化

#### 第 2 周：做出一单完整订单

目标：

- 1 位客人入场
- 显示目标波形和文字描述
- 可调酒、提交、结算
- 显示评价和匹配度

验收标准：

- 能完整跑通 `接单 -> 调酒 -> 提交 -> 评价`
- 失败和成功有明确反馈差异

#### 第 3 周：做出一天闭环

目标：

- 连续接待 `5` 单
- 有资金 / 电力 / 评分
- 有日结与失败判定

验收标准：

- 单次游玩达到 `8-10` 分钟
- 玩家会因为资源压力主动调整打法

#### 第 4 周：做出成长动力

目标：

- 商店购买
- 3 个升级项生效
- 自动存档

验收标准：

- 玩家完成一天后有明确“下一天更强一点”的预期

#### 第 5 周：做出内容差异

目标：

- 3 位客人
- 3 类事件
- 9 个订单模板

### 19.13 阶段验收表

为了避免“做了很多，但不知道是否算完成”，每个阶段都用下表判断：

| 阶段 | 必须成立的问题 | 不成立时的处理 |
| --- | --- | --- |
| 可操作酒杯 | 玩家能否看懂操作对波形的影响 | 先改反馈，不加内容 |
| 单单闭环 | 玩家是否愿意为了更高分重复尝试 | 先改匹配反馈，不加剧情 |
| 单日闭环 | 玩家是否感到经营压力真实存在 | 先改数值，不加新系统 |
| 成长闭环 | 玩家过完一天后是否想进下一天 | 先改奖励结构，不加新客人 |
| 内容扩张 | 玩家是否能分辨不同客人和订单风格 | 先改模板差异，不加更多数量 |

### 19.14 明确暂缓开发清单

### 19.16 最终执行建议

这个项目最值得赌的不是“内容量”，而是下面这个体验是否真的成立：

`我正在用调酒动作修正一个人的情绪频率，而且这件事同时影响故事和生存。`

所以实际开发时的优先级应该始终固定为：

1. 波形反馈是否好懂
2. 调酒动作是否有手感
3. 经营资源是否形成压力
4. 客人是否有记忆点
5. 扩展系统是否值得加入

只要顺序不乱，这个方案不但能做，而且很有机会做出独特气质。
