# 共振酒吧 MVP 开发任务拆解清单

这份清单基于项目计划书的“执行优化”章节，将 6 周路线图和模块开发顺序拆解为可以直接分配的开发任务。建议配合 Vite + TypeScript 工程逐项销账。

## 阶段 0：工程初始化与核心状态定义（约 1-2 天）

- [ ] **0.1 搭建前端基础工程**
  - [ ] 初始化 `Vite + TypeScript` 纯净工程（无 React/Vue 强依赖，或按需引入轻量状态库）。
  - [ ] 按照计划书 19.10 创建基础目录结构（`game/`, `systems/`, `content/`, `ui/`, `audio/`）。
  - [ ] 引入并配置 `Canvas` 画布环境和自适应缩放容器。
- [ ] **0.2 定义核心数据类型 (TS Interfaces)**
  - [ ] 根据数据表模板，定义 `DrinkState`, `MixAction`, `GuestOrder`, `WaveParams`。
  - [ ] 定义游戏核心全局状态 `GameState`（资金、电力、评分、天数）。
  - [ ] 定义单单流程状态机枚举 `OrderFlowState`。

## 阶段 1：可操作酒杯与波形反馈（约 3-5 天）

- [ ] **1.1 实现调酒台基础输入 (UI 占位)**
  - [ ] 渲染 3 种基酒按钮。
  - [ ] 渲染 5 种操作按钮（倒酒长按、加糖浆、加冰块、搅拌、加苏打水）。
  - [ ] 渲染 提交（上酒）和 重做 按钮。
- [ ] **1.2 实现 `mixology-system`（液体状态机）**
  - [ ] 编写核心函数 `applyAction(state: DrinkState, action: MixAction): DrinkState`。
  - [ ] 实现动作对三大核心属性（烈度、甜度、温度）及容量（volume）、气泡感（sparkle）的数值计算公式。
- [ ] **1.3 实现 `wave-generator`（波形生成核心）**
  - [ ] 将三大属性（烈度、甜度、温度）映射到波形的物理参数（振幅、频率、衰减/相位）。
  - [ ] 实现一个生成长度为 128 或 256 的波形采样数组的纯函数 `generateWave(params): number[]`。
- [ ] **1.4 实时视觉反馈 (Canvas)**
  - [ ] 用 Canvas 实时绘制当前液体的波形曲线。
  - [ ] 简易绘制当前杯中液体的颜色/容量百分比。
  - [ ] **阶段测试点**：点击操作按钮，右侧波形和数值必须能看到明显的肉眼变化。

## 阶段 2：单单流程与结算匹配（约 3-4 天）

- [ ] **2.1 实现单客流程状态机**
  - [ ] 实现主循环，处理 `guest_enter` -> `dialogue` -> `mixing` -> `result` 的状态流转。
- [ ] **2.2 接入目标订单**
  - [ ] 预填 3 个订单模板，并能在 UI 上展示客人的“目标波形（红色）”与“情绪描述文本”。
- [ ] **2.3 实现 `wave-match` 算法**
  - [ ] 编写函数比较 目标波形数组 与 玩家波形数组 的逐点均方误差 (MSE)。
  - [ ] 将误差转换为 `0-100` 的最终匹配度得分（包含溢杯、重做惩罚）。
- [ ] **2.4 结果结算面板**
  - [ ] 在 `result` 状态，展示评价（完美共振/很满意/失望等）、得分。
  - [ ] **阶段测试点**：玩家可以完整接待一名客人，并能通过刻意调对或调错，看到匹配度和结算文本的改变。

## 阶段 3：单日闭环与经营压力（约 2-3 天）

- [ ] **3.1 接入经营三维资源**
  - [ ] 初始化资金（120）、电力（18）、评分（40）。
  - [ ] 订单结算后，将得分转化为资金收益、基础电力消耗、评分增减。
  - [ ] UI 上方实时显示三维资源进度条。
- [ ] **3.2 单日循环（5 单制）**
  - [ ] 设定 `ordersPerDay = 5`，完成 5 个客人后进入 `resource_settlement`（日结）状态。
  - [ ] 每日开场扣除房租，日结计算耗电，并恢复基础电力。
- [ ] **3.3 失败判定与 Game Over**
  - [ ] 任何资源归零时，触发 Game Over 流程及结算面板。
  - [ ] **阶段测试点**：玩家可以连续玩完一天，乱玩会破产/停电，认真玩能活到第二天。

## 阶段 4：商店、成长与持久化（约 2-3 天）

- [ ] **4.1 商店系统与 UI**
  - [ ] 日结后进入商店页，显示 6 个可购买升级项。
  - [ ] 扣除资金，将升级效果写入 `GameState`（如：容量上限提升、基础补电提升）。
- [ ] **4.2 本地自动存档 `save-system`**
  - [ ] 每天结算后，将整个 `GameState` 序列化为 JSON 写入 `localStorage`。
  - [ ] 游戏启动时，若检测到存档，提供“继续营业”选项。
- [ ] **4.3 随机事件（MVP：3种）**
  - [ ] 每天开始时从事件池抽取 1 个事件（如下雨、冰块短缺），并全屏提示。
  - [ ] 将事件影响（如数值修正系数）注入到当天的 `mixology-system` 中。

## 阶段 5：基础视听与打磨（约 2-4 天）

- [ ] **5.1 基础音效接入 (`Web Audio`)**
  - [ ] 倒酒流体声、冰块碰撞声、UI 点击声。
  - [ ] 结算成功（和弦）与失败（失真）的提示音。
- [ ] **5.2 美术占位符替换**
  - [ ] 使用色块或简单的像素图替换所有占位 UI。
  - [ ] 波形曲线增加赛博朋克风的“霓虹发光”滤镜（Canvas `shadowBlur`）。
- [ ] **5.3 试玩与数值微调**
  - [ ] 调节 3 种客人的初始难度，确保第一单不看说明也能 80 分以上。
  - [ ] 调节资金消耗速度，确保玩家在第 3 天左右会面临购买设备还是留钱交租的纠结。
