# 角色档案与多轮对话扩充计划

## 1. 目标与背景 (Summary)

根据老陈（mechanic\_01）已经实现的多轮对话数据格式（`DialogueTree`），为剩余的 6 位角色（Z3R0、雷队、Sarah、Miku、Unit-7、Jax）扩充对应风格的多轮交互对话。同时，对齐所有角色的档案（archives）解锁阈值，并确保新增的数据结构与当前系统的 `DialogueEntry` 接口完全兼容。

## 2. 现状分析 (Current State Analysis)

* **对话格式**：目前除了老陈在 `neutral` 阶段有一套完整的 `DialogueTree` 之外，其他角色的 `affinityLevels` 仍然全是单句字符串。

* **档案阈值**：系统中 `exec_01` (Sarah) 和 `robot_01` (Unit-7) 的档案解锁阈值为 `25/55/85`，不符合核心记忆中统一的 `20/50/80` 规范。

* **接口兼容性**：现有的 `DialogueEntry` 类型定义为 `string | DialogueTree`，HUD 和 NPC 分发系统已经支持此联合类型。只要确保新增的数据严格遵循 `DialogueTree` 接口（`rootId`, `nodes`, `options`, 且结束选项不带 `nextId`），就不会有兼容性问题。

## 3. 具体修改方案 (Proposed Changes)

**目标文件：** `src/content/guests.ts`

### 3.1 统一档案阈值

* 将 `exec_01` (Sarah) 的 archives threshold 从 `25/55/85` 修改为 `20/50/80`。

* 将 `robot_01` (Unit-7) 的 archives threshold 从 `25/55/85` 修改为 `20/50/80`。

### 3.2 扩充多轮对话 (DialogueTree)

为每位角色在其 `neutral` 或 `friendly` 池中至少加入一个符合其设定的多轮对话树：

1. **hacker\_01 (Z3R0)**

   * **设定**：数字咨询师，社恐，面临数字焦虑。

   * **对话树主题**：关于最近一次黑客行动的意外发现或面对断网的恐慌，选项允许玩家表现出关心或冷漠。
2. **cop\_01 (雷队)**

   * **设定**：社区协警，看重人情味，疲于应付系统指令。

   * **对话树主题**：关于街区小贩违章或巡逻时的两难抉择，玩家可以选择建议他按规矩办事或遵循人情。
3. **exec\_01 (Sarah)**

   * **设定**：高级执行官，厌倦高压与完美，向往旧时光。

   * **对话树主题**：关于公司裁员计划带来的负罪感，或讨论某张旧黑胶唱片，玩家的回答影响好感度。
4. **idol\_01 (Miku)**

   * **设定**：地下歌手，坚持真唱，对抗 AI 娱乐工业。

   * **对话树主题**：关于声音受损的担忧或对 AI 假唱的愤慨，玩家可以选择鼓励她坚持或建议妥协。
5. **robot\_01 (Unit-7)**

   * **设定**：旧型服务机器人，产生逻辑溢出，探讨存在主义。

   * **对话树主题**：询问玩家关于“记忆删除”或“灵魂”的定义，选项可以偏向理性逻辑或感性认知。
6. **drifter\_01 (Jax)**

   * **设定**：街头浪人，曾是飞行员，向往自由之地。

   * **对话树主题**：分享一件从垃圾场捡到的旧物，或讨论高空俯瞰城市的压抑感，玩家可以询问过去或专注于当前。

### 3.3 接口格式对齐

每个 `DialogueTree` 必须包含：

* `rootId`: 初始节点的 ID。

* `nodes`: 包含所有对话节点的 Record。

  * `id`: 节点 ID。

  * `text`: NPC 的台词。

  * `options`: 玩家的选项数组。

    * `text`: 选项文本。

    * `nextId` (可选): 跳转节点，若未定义则直接结束对话进入调酒。

    * `affinityChange` (可选): 影响好感度。

## 4. 假设与决策 (Assumptions & Decisions)

* **对话位置**：为了让玩家更容易触发，多轮对话将主要添加在 `neutral` 和 `friendly` 层级中。

* **好感度调整**：选项中的 `affinityChange` 将适度使用（例如 +5 或 -2），以体现玩家选择的 RPG 扮演感。

* **文案风格**：遵循核心记忆中的“混合递进型”规范，去 IP 化，避免过于黑暗敏感的话题，保持“轻赛博”的日常感。

## 5. 验证步骤 (Verification steps)

1. 运行 `npm run typecheck`，确保新增的数据完全符合 TypeScript 接口定义。
2. 启动开发服务器 `npm run dev`，并在浏览器中随机接待几位客人，验证能否正常渲染选项按钮，且点击末尾选项后能顺利进入 `mixing_view` 调酒界面。
