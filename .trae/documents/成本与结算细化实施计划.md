# 成本与结算细化实施计划

## Summary
- 目标：把当前“简化资源结算”升级为“按消耗记账 + 按日汇总 + 单后评分结算 + 双时机失败判定”的稳定经营闭环。
- 成功标准：
  - 资金按单实时扣除基酒/添加剂成本，订单完成后按“固定出杯价 + 评分加成”入账。
  - 电力仅由冰块消耗（每块固定扣电）。
  - 评分仅由每单客评变动，并在“单后 + 日结”两次判定是否归零。
  - 任一资源在当天任意时刻 `<= 0` 触发 `game_over`。
  - 难度曲线满足“前 3 天更宽松，7 天后明显变难”。

## Current State Analysis
- `src/systems/economy/economy-system.ts`
  - 当前是按分数直接加钱、减电、改评分，尚未区分“成本扣除”和“订单收入”。
  - 日结已有固定租金雏形，但未分梯度，也未基于“当天真实消耗”出账单。
- `src/game/game-loop.ts`
  - 订单提交时会走评分与资源变更，但消耗来源没有账本（无法回溯每单用料成本）。
  - 有 `resource_settlement`、`game_over` 流程点位，可直接承载新规则。
- `src/systems/mixology/mixology-system.ts`
  - 已能识别动作和原料 ID（`vodka/gin/whisky/simple_syrup/lemon_juice/soda_water/ice_cube`），具备做实时扣费/扣电的触发点。
- `src/content/spirits.ts` 与 `src/content/ingredients.ts`
  - 原料数据齐全，但没有经济字段（如 `unitCost`、`powerCost`）。
- `src/ui/hud-renderer.ts`
  - 面板已显示资金/电力/评分，可扩展“当日成本/当日收入/净收益”展示。

## Proposed Changes

### 1) 数据模型扩展（成本账本与难度）
- 文件：`src/game/game-state.ts`
- 变更：
  - 新增 `dailyLedger`：
    - `ingredientCostToday: number`
    - `orderIncomeToday: number`
    - `rentToday: number`
    - `powerFromIceToday: number`
    - `ordersToday: number`
  - 新增 `balanceConfig`（可调参数）：
    - `baseOrderPrice: number`
    - `scoreBonusTable`
    - `rentByDay(day): number`
    - `startingResourcesByDayTier`（用于新开局默认值）
- 原因：后续每单扣费、日结账单、调参都依赖稳定账本与参数中心。

### 2) 原料经济字段标准化
- 文件：`src/content/spirits.ts`、`src/content/ingredients.ts`
- 变更：
  - 给每个可消耗原料增加：
    - `unitCost`（资金成本）
    - `powerCost`（仅冰块为 1，其它为 0）
  - 参考首轮建议：
    - 基酒：`vodka=9, gin=8, whisky=10`
    - 添加剂：`simple_syrup=2, lemon_juice=3, soda_water=3`
    - 冰块：`ice_cube=1, powerCost=1`
- 原因：把成本责任从逻辑硬编码迁移到数据层，便于平衡。

### 3) 结算系统重构（按规则拆分）
- 文件：`src/systems/economy/economy-system.ts`
- 变更：
  - 新增 `applyIngredientCost(state, ingredientId)`：
    - 实时扣 `money -= unitCost`
    - 若为冰块则 `power -= powerCost`
    - 累加 `dailyLedger`
  - 新增 `applyOrderIncome(state, score)`：
    - 收入 = `baseOrderPrice + scoreBonus(score)`
    - `money += income`
    - 记录 `orderIncomeToday`
  - 新增 `applyOrderRating(state, score)`：
    - 仅改 `rating`，不改 money/power
  - 改造 `applyDailySettlement(state)`：
    - 按天数分梯度租金：
      - Day 1-3：低租金
      - Day 4-7：中租金
      - Day 8+：高租金
    - `money -= rent`
    - 清零/归档 `dailyLedger`
  - 统一 `checkGameOver(state)`：
    - 任一资源 `<= 0` 返回 `true`
- 原因：严格落地你定义的三资源规则与失败条件。

### 4) 动作触发点接入成本与电力
- 文件：`src/game/game-loop.ts`、`src/systems/mixology/mixology-system.ts`
- 变更：
  - 在拖拽释放成功并执行动作前，识别动作映射到原料 ID。
  - 调用 `applyIngredientCost()` 实时扣费/扣电。
  - 若扣减后触发 `checkGameOver`，立刻切到 `game_over`，阻止后续动作。
- 原因：保证“实时消耗”真实发生，不等到提交才扣。

### 5) 单后结算只处理收入与评分
- 文件：`src/game/game-loop.ts`
- 变更：
  - `submit` 分支改为：
    1. 计算匹配分
    2. `applyOrderIncome(state, score)`
    3. `applyOrderRating(state, score)`
    4. 单后 `checkGameOver(state)`（评分可能归零）
  - 保留现有结果面板流转。
- 原因：与你的规则保持一致：评分来自客评，资金主要由“成本+出杯收入”驱动。

### 6) 日结与难度曲线
- 文件：`src/game/game-loop.ts`、`src/systems/economy/economy-system.ts`
- 变更：
  - `next_day` 中执行：
    - `applyDailySettlement()`
    - 日结后再次 `checkGameOver()`（双时机判定）
  - 难度曲线参数：
    - 初始资源建议：`money=180, power=24, rating=55`（前 3 天更稳）
    - Day 1-3：租金低、评分扣减系数低
    - Day 4-7：租金中、评分波动恢复默认
    - Day 8+：租金高、低分罚分加重（仅系数，不改核心规则）
- 原因：满足“前三天简单，第七天后更难”。

### 7) UI 账单与反馈补充
- 文件：`src/ui/hud-renderer.ts`
- 变更：
  - 状态栏新增：
    - `今日耗材`
    - `今日收入`
    - `今日净收益`
    - `今日冰耗电`
  - `resource_settlement` 面板显示分项账单（成本、收入、租金、净变化）。
- 原因：玩家能理解“为什么破产/断电/掉分”，提升可玩性与可调试性。

### 8) 存档一致性
- 文件：`src/systems/save/save-system.ts`、`src/main.ts`
- 变更：
  - 升级存档版本号，纳入 `dailyLedger` 与新配置字段。
  - `main.ts` 启动优先读取存档，失败时回退默认状态。
- 原因：新增结算结构后防止旧档异常。

## Assumptions & Decisions
- 已锁定决策：
  - 资金：按单实时扣耗材，提交后按“固定出杯价 + 评分加成”入账。
  - 电力：仅冰块消耗（每块固定扣电）。
  - 评分：仅订单客评变动。
  - 失败：单后与日结都判定，任一资源 `<= 0` 当天立即结束。
  - 日结：保留固定租金，且按天数分梯度。
  - 难度：前 3 天宽松，7 天后提升压力。
- 计划内不包含：
  - 新增剧情/新美术资源/新操作道具，仅改结算与数值闭环。

## Verification Steps
- 功能验证：
  1. 开始新一天，拖入 1 次基酒 + 2 次添加剂 + 1 块冰，检查资金与电力即时下降，且账本累加正确。
  2. 提交订单后，检查仅“收入+评分”变化；评分由匹配度决定。
  3. 当评分在单后跌至 0，立即进入 `game_over`。
  4. 订单达到上限触发日结，显示“收入/耗材/租金/净收益”，并执行租金扣减。
  5. 日结后任一资源归零，进入 `game_over`。
- 难度验证：
  1. Day 1-3 平均可稳定续航（不频繁破产）。
  2. Day 8+ 在相同操作水平下明显更容易出现资源压力。
- 工程验证：
  - `npm run typecheck`
  - `npm run build`
