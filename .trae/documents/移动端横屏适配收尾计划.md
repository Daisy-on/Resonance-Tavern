# 移动端横屏适配收尾计划

## Summary

本次只适配**移动端横屏**，不做竖屏完整玩法布局。目标分为两层：

1. 完成已经开始但尚未收尾的移动端横屏适配，把 `mixing_view`、HUD 和输入体系统一到同一套横屏规则。
2. 明确消除“交互元素被遮挡”风险，保证移动端横屏下所有可点击、可拖拽、可提交的元素都处于可见且可操作状态。

用户已确认的约束如下：

- 竖屏时只提示“建议横屏体验”，不强制阻断。
- 调酒输入统一走 Pointer 事件。
- 为了避免遮挡，可以裁掉部分底部吧台装饰区域。
- 必须完整显示且不被遮挡的核心区域包括：示波器主屏、特殊事件/客人需求、实时提示、调酒工具与杯子落点。

## Proposed Changes

### 1. 清理 `mixing_view` 中残留的 DOM 交互控件

**文件：** `src/ui/hud-renderer.ts`, `src/styles.css`

**修改内容**

- 删除 `#table-controls` 的模板输出、显示控制和相关按钮启停逻辑。
- 删除 `.table-btn`、`#table-controls` 的样式与横屏媒体查询规则。
- 保留调酒阶段真正需要的 DOM 交互入口，仅保留不会压住画布操作区的内容：
  - 竖屏提示层
  - 左上状态抽屉
  - 对话面板
  - 结果面板
  - 档案弹层

**原因**

- 当前项目的调酒操作主交互已经在 Canvas 中完成，残留的 `table-controls` 会与“操作区不可遮挡”的目标冲突。
- 这一步同时收敛了此前 UI 优化要求，避免移动端适配把旧按钮重新带回来。

### 2. 把 HUD 响应式规则收敛为“横屏手机优先”

**文件：** `src/styles.css`, `src/ui/hud-renderer.ts`

**修改内容**

- 以 `@media (max-width: 932px) and (orientation: landscape)` 为主规则继续收束移动端横屏 HUD。
- 保留竖屏提示规则，但不再为竖屏提供完整可玩布局，只确保提示可见且不妨碍浏览。
- 调整各 HUD 元素的边界与层级，保证不压到核心 Canvas 区域：
  - `#status-panel` 固定在左上 safe area 内，限制展开高度并允许内部滚动。
  - `#dialogue-panel` 仅在 `guest_enter / dialogue / mixing` 出现，移动到左下或底部边角，不进入 `mixing_view`。
  - `#result-panel` 控制最大宽高，确保小高度横屏下仍完整可见并可滚动。
  - `#orientation-hint` 仅在粗指针竖屏显示，保持 `pointer-events: none`。

**原因**

- 当前主要矛盾不是宽度不足，而是横屏手机高度过矮。
- DOM HUD 只能占边角，不能继续侵入示波器和调酒桌面。

### 3. 完成 `mixing_view` 布局参数的最终校准

**文件：** `src/ui/mixing-layout.ts`, `src/ui/bar-renderer.ts`

**修改内容**

- 继续以 `getMixingViewLayout(width, height)` 作为唯一布局来源，不新增第二套移动端布局逻辑。
- 执行阶段重点微调以下参数，形成稳定的横屏手机布局：
  - `tableHeight` 和 `tableY`
  - `propY`
  - `waveArea.width / height / centerY`
  - `infoBox.width / height / y`
  - `cup.x / y / dropRect`
  - `stirToolsRect` 与 `advancedToolsRect`
- 明确紧凑横屏的视觉策略：
  - 允许下裁桌面装饰区
  - 优先让示波器、信息框、实时提示、拖拽工具完整露出
  - 保持左右操作分区与杯子投放区之间有足够空隙，避免误触

**原因**

- 当前框架已经对，但参数仍是第一次压缩版，必须根据目标机型做收口。
- 这一步决定“看得见”和“点得到”是否真正成立。

### 4. 校准共享布局与命中区切分的一致性

**文件：** `src/game/game-loop.ts`, `src/ui/mixing-layout.ts`, `src/ui/bar-renderer.ts`

**修改内容**

- 保持命中检测继续读取 `getMixingViewLayout()`。
- 对现有固定切分阈值做一次系统化校准，确保与实际渲染顺序和图元宽度一致：
  - 基酒四个瓶子的分段阈值
  - 添加剂五个项目的分段阈值
  - 搅拌工具左右切分
  - 高级工具三段切分
- 如有必要，把“命中区矩形 + 分段宽度”也抽进 `mixing-layout.ts`，避免渲染和交互再次靠手工同步。

**原因**

- 目前已经从“完全双份布局”进步到“共享大框架”，但子分段仍依赖固定数值。
- 横屏收缩后最容易出现的回归就是图元位置变了，而切分边界没跟上。

### 5. 完整验证 Pointer 交互在移动端横屏下的拖拽闭环

**文件：** `src/game/game-loop.ts`, `src/styles.css`

**修改内容**

- 保持 Pointer 事件为唯一调酒输入通道：
  - `pointerdown`
  - `pointermove`
  - `pointerup`
  - `pointercancel`
- 保持 `canvas { touch-action: none; }`。
- 执行阶段重点验证以下细节，必要时再做轻量修补：
  - `setPointerCapture` / `releasePointerCapture` 在移动端浏览器下不会造成异常
  - 拖拽出杯子区域后取消释放能正确清空状态
  - 未解锁道具不会被拖起
  - 横屏切换和 resize 后命中区仍然准确

**原因**

- Pointer 改造已完成，但需要以移动端真实场景口径验证，而不是只看类型检查通过。

### 6. 用真实目标尺寸做验收，不再只依赖桌面直觉

**文件：** 无代码新增，执行阶段通过现有项目验证

**修改内容**

- 验证至少覆盖以下横屏尺寸：
  - `667x375`
  - `812x375`
  - `844x390`
  - `915x412`
- 每个尺寸都按同一口径检查：
  - 示波器是否完整显示
  - 特殊事件 / 客人需求是否完整显示
  - 实时提示是否完整显示
  - 左右道具区是否完整显示
  - 杯子及落杯区是否完整显示
  - 左上状态抽屉展开后是否压到关键交互
  - 结果面板和档案弹层是否能完整操作

**原因**

- 这次需求的成败标准是“移动端横屏真正能玩”，不是“桌面浏览器缩窄后大致像样”。

## Assumptions & Decisions

- 不新增横屏锁定逻辑，不强制用户旋转屏幕。
- 不为竖屏提供完整调酒操作适配，只保留提示与最低限度浏览能力。
- 不恢复右下角汇总面板在 `mixing_view` 中的显示。
- 不保留旧的 `table-controls` 浮层按钮，避免与 Canvas 调酒交互形成双入口。
- 若需要在视觉与可操作性之间取舍，优先保证可操作性。
- 若某些底部桌面装饰必须被裁掉，视为可接受。

## Verification Steps

1. 运行 `npm run typecheck`，确认布局参数、HUD 清理和 Pointer 交互调整无类型错误。
2. 运行 `npm run build`，确认 Vite 构建与样式收敛后没有打包问题。
3. 在横屏模拟尺寸 `667x375 / 812x375 / 844x390 / 915x412` 下逐项验证：
   - `mixing_view` 中示波器、信息框、提示、工具区和杯子全部可见。
   - 任何可交互元素都没有被 DOM 面板遮挡。
4. 验证调酒拖拽：
   - 基酒、添加剂、工具都能正常拖起。
   - 拖到杯子内可成功生效。
   - 未解锁物品不可拖起。
   - 拖拽取消后状态可恢复。
5. 验证对话与结算：
   - 竖屏时提示出现但不阻断。
   - `guest_enter` 的对话按钮和选项可点击。
   - 结果面板按钮、共振码按钮、求救按钮不超出屏幕。
   - 档案弹层可打开、滚动、关闭。
6. 旧存档回归验证：
   - 加载旧存档不会因 `lastScore` 为 `null` 造成结果面板崩溃。

## Execution Order

1. 清理 `table-controls` 相关模板、逻辑和样式。
2. 收束 `styles.css` 的移动端横屏规则，去掉与当前目标冲突的旧规则。
3. 微调 `mixing-layout.ts` 的紧凑横屏参数，并同步校准 `bar-renderer.ts`。
4. 校准 `game-loop.ts` 的命中区切分边界，必要时继续抽象进布局文件。
5. 运行 `typecheck/build`。
6. 做横屏多尺寸验证并按结果做最后一轮细调。
