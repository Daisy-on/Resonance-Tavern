# 波形参数直觉化与难度重构计划

## Summary

* 目标：解决“玩家难以理解工具在改什么、前中后期波形难度不成层、现有工具集无法稳定还原客人波形”这三个核心玩法问题。

* 本次重构采用“玩家直接可见的波形参数体系”，不再把核心操作建立在“酸度/甜度/混合度”等抽象调酒术语上。

* 成功标准：

  * Day 1-2 的订单只要求玩家控制 `基础形状 / 振幅 / 周期 / 相位` 四个维度，且现有工具可稳定完成。

  * Day 3-7 才逐步引入少量“尖锐度/锯齿/噪声”类高级维度，难度增长清晰且可学习。

  * Day 8+ 再加入组合复杂度，并补足“抚平锯齿/降低毛刺”的独立工具，不再让冰块承担不合理的复合职责。

  * UI 中每个工具的作用能直接映射到可见波形属性，玩家无需猜测“这个抽象参数到底改了什么”。

## Current State Analysis

* 当前波形参数由 [wave-generator.ts](file:///d:/Homework_vs/game-pr/src/systems/wave/wave-generator.ts) 中的 `amplitude/frequency/decay/noise/symmetry/phase/harmonics` 组成，但前端并未把这些维度转成玩家可理解的明确操作语义。

* 当前动作效果定义集中在 [mixology-system.ts](file:///d:/Homework_vs/game-pr/src/systems/mixology/mixology-system.ts)，但很多动作仍以“甜度/酸度/混合度”等内部量累计，再间接映射到波形，导致玩家难以建立直觉。

* 当前基础工具中：

  * 基酒决定基础波形类型与部分振幅基底。

  * 糖浆通过 `sweetness` 推高频率。

  * 苏打通过 `sparkle` 增加毛刺。

  * “捣拌棒”（实际是 `tonic_essence`）通过负甜度降低频率。

  * 顺/逆搅拌调整相位。

  * 冰块同时影响温度、容量、稀释、烈度，职责过重。

* 当前订单难度在 [npc-system.ts](file:///d:/Homework_vs/game-pr/src/systems/npc/npc-system.ts) 里主要靠 `difficulty + allowVariance + 参数随机扰动` 实现，并不是明确的“波形家族分层”。

* 当前评分在 [wave-match.ts](file:///d:/Homework_vs/game-pr/src/systems/wave/wave-match.ts) 里已具备分项评分，但分项名称依然偏技术化，没有与玩家手里的工具和学习曲线绑定。

* 产品文档 [cyber-resonance-bar-plan.md](file:///d:/Homework_vs/game-pr/docs/cyber-resonance-bar-plan.md) 已提出“新手期 1-2 天单峰波形、3-7 天多参数组合、8 天后高压复杂波形”的大方向，但代码尚未重构到这个层次。

## Proposed Changes

### 1) 重构为“玩家可见波形参数体系”

* 文件：

  * `src/systems/mixology/drink-state.ts`

  * `src/systems/wave/wave-generator.ts`

  * `src/ui/hud-renderer.ts`

* 变更：

  * 将调酒内部状态收敛到玩家可理解的 6 个主维度：

    * `baseShape`：基础形状

    * `amplitude`：振幅

    * `periodLevel`：周期档位

    * `phaseStep`：相位档位

    * `edgeSharpness`：边缘尖锐度/锯齿感

    * `noiseLevel`：毛刺/噪声

  * 将现有 `sweetness/acidity/blend/oxidation` 从“玩家直接体验层”降级为内部兼容字段，逐步移出调酒主逻辑；本轮计划中 UI 与工具不再直接使用这些术语。

  * 在 HUD 中直接显示上述 6 个维度的当前档位或方向，不再显示“酸度/甜度”类抽象词。

* 原因：

  * 用户已明确要求“改成直接参数体系”，并指出“酸度是什么”对波形变化过于模糊。

### 2) 重新定义现有工具的最小职责表（修正版）

*   文件：
    *   `src/systems/mixology/mix-actions.ts`
    *   `src/systems/mixology/mixology-system.ts`
    *   `src/content/spirits.ts`
    *   `src/content/ingredients.ts`
    *   `src/ui/hud-renderer.ts`
    *   `src/ui/bar-renderer.ts`

*   变更原则：
    *   “一个道具只能有一种主效果，副作用必须极微弱且不能影响核心波形还原”。
    *   删除重复的工具（删除压碎棒/muddle，其功能由其他工具替代）。

*   工具与波形属性映射表（严格执行）：
    *   **基酒 (伏特加/琴酒/威士忌/朗姆)**：保留切换基础波形家族，并提供初始的振幅和周期基底。
    *   **糖浆**：主控 **减小周期 (增大频率)**。副作用：极微弱的振幅增加。
    *   **冰块**：主控 **减小振幅 (压低波形)**。副作用：极微弱的衰减变化。
    *   **捣拌棒**：主控 **增大周期 (减小频率)**。拉长波形。无副作用。
    *   **顺/逆时针搅拌**：主控 **相位平移 (+/-)**。无副作用。
    *   **柠檬**：（基于当前代码 `ingredients.ts` 评估）主控 **不对称偏斜 (EdgeSharpness+)**。视觉上会让原本对称的山峰向一侧倾斜，产生锋利的边缘感。属于 **波形形状修改**。
    *   **苏打水**：主控 **引入锯齿噪音 (Noise+)**。视觉上让波形线条变得粗糙抖动。属于 **波形形状修改**。

    *   **高阶工具明确定位（必须明确影响波形的方式）：**
        *   **苦精 (Bitters)**：属于 **特殊效果（不改变波形视觉形状）**。作用是“放宽相位容错判定”。在最终结算时，暗中扩大波形对齐的判定窗口（Shift Window），允许玩家的波形在左右有一定偏差的情况下依然获得高分，是拯救手残的辅助道具。
        *   **摇壶 (Shaker)**：属于 **波形形状修改**。主控 **谐波叠加 (Harmonics+)**。视觉上会在主波形上叠加出次级波纹，形成双峰、多峰或复杂的复合波形。用于还原后期客人的复杂重叠波。
        *   **量杯 (原精确注入器)**：属于 **波形形状修改（平滑器）**。主控 **抚平毛刺与偏斜 (Noise-, EdgeSharpness-)**。视觉上像是一个滤波器，强制抚平波形上的高频抖动和不对称，将波形还原回干净平滑的曲线。
        *   **喷枪 (Blowtorch)**：属于 **波形形状修改（拖尾控制）**。主控 **减小拖尾衰减 (Decay-)**。视觉上改变波形的右侧包络线，使得原本向右逐渐变矮消失的波形变得饱满，让波形在屏幕右侧依然保持较高的高度。

*   原因：
    *   用户明确要求：必须清晰界定所有高级工具到底是改变波形形状还是引入特殊效果。这一版定义彻底消除了工具之间的功能重叠，确保每个工具都有其不可替代的特定作用域。

### 3) 按天数重构波形难度层

* 文件：

  * `src/content/orders.ts`

  * `src/systems/npc/npc-system.ts`

  * `src/game/unlock-system.ts`

  * `src/game/game-state.ts`

* 变更：

  * 将订单拆为三层波形家族，而不是只靠 `difficulty` 数字：

    * Day 1-2：基础波阶段

      * 只生成 `sine / triangle / square`。

      * 仅出现 `振幅 / 周期 / 相位` 差异。

      * 不出现锯齿、噪声、复杂谐波。

    * Day 3-7：简单进阶阶段

      * 开始引入轻量 `edgeSharpness` 或轻量 `noiseLevel`。

      * 但同一订单最多只新增 1 个高级维度，避免一次性堆难度。

      * 订单文案要明确提示“更锋利”“更平滑”“略带毛刺”等直白描述。

    * Day 8+：高压组合阶段

      * 允许 `基础形状 + 振幅 + 周期 + 相位 + 锯齿/噪声` 的双高级维度组合。

      * 引入“需要抚平锯齿”的目标，配套启用平滑工具。

  * 继续沿用现有解锁节奏，但把 Day 3 作为高级维度开放的起点，Day 8 作为复杂组合开放起点。

* 原因：

  * 用户明确要求：前两天只练基础形状，第三天起才加入简单新样式，第八天后才明显变难。

### 4) 重写波形生成映射，避免“抽象词间接控制”

* 原因：

  * 当前“柠檬提升酸度”这类间接映射太抽象；重构后每个维度都应可独立教学和独立观察。

## Assumptions & Decisions

* 已锁定决策：

  * 电力额度调整到 30。

  * 调酒主体验改成“玩家直接可见的波形参数体系”。

  * 前两天只训练 `基础形状 / 振幅 / 周期 / 相位`。

  * 现有工具优先复用，只重写职责，不盲目新增大批新道具。

  * 第 8 天开始必须有独立“抚平锯齿/降低毛刺”的工具。

* 兼容策略：

  * 内部旧字段可暂时保留作为过渡兼容层，但 UI、订单和动作逻辑全部切到新参数体系。

  * 旧订单数据可通过默认值迁移到新的维度结构。

* 本轮不做的事：

  * 不引入全新剧情系统。

  * 不引入新的经营资源种类。

  * 不扩展到 15+ 无限模式，只先把 1-8+ 的清晰学习曲线做好。
