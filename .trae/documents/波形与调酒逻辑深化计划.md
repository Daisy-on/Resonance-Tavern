# 波形与调酒逻辑深化计划

## Summary
- 目标：解决“道具无法还原目标波形、拟合评分体感不准、后期复杂度与道具成长不足”三个核心问题。
- 成功标准：
  - 玩家可通过现有/新增道具稳定调节 `振幅(amplitude)`、`频率(frequency)`、`相位(phase)` 三大维度。
  - 同一杯酒在视觉上“明显接近”目标时，评分显著提升，减少“看起来很像却分低”。
  - 随天数推进，目标波形复杂度分层增加，并按天数阈值自动解锁更多基酒与道具。
  - 新增道具有对应像素素材，并在调酒桌布局中合理显示与可交互。

## Current State Analysis
- 波形参数仅有 `amplitude/frequency/decay/noise/symmetry/baseShape`，缺少相位控制入口：
  - `src/systems/wave/wave-generator.ts`
- 拟合算法是 `MSE + 参数差加权`，未做相位错位容忍，导致视觉近似但分数偏低：
  - `src/systems/wave/wave-match.ts`
- 调酒动作对属性影响偏“加法”，且动作映射分散在多个模块：
  - `src/systems/mixology/mixology-system.ts`
  - `src/game/game-loop.ts`
- 订单复杂度仅做少量随机波动，尚未引入明确的高复杂波形家族：
  - `src/systems/npc/npc-system.ts`
  - `src/content/orders.ts`
- 调酒桌道具渲染已具备像素绘制基础，但道具种类与解锁联动不足：
  - `src/ui/bar-renderer.ts`
  - `src/ui/pixel-assets.ts`

## Proposed Changes

### 1) 统一“波形可控维度”数据模型
- 文件：
  - `src/systems/wave/wave-generator.ts`
  - `src/systems/mixology/drink-state.ts`
  - `src/content/orders.ts`
- 变更：
  - 扩展 `WaveParams` 增加 `phase`（相位偏移）与 `harmonics`（谐波复杂度因子）。
  - 在 `DrinkState` 增加 `phaseOffset` 字段，作为调酒过程中可累积控制量。
  - 订单 `targetParams` 增加 `phase` 与 `harmonics`，支持后期复杂波形目标。
- 原因：先把“可调”和“要匹配”的参数空间对齐，避免先天不可达。

### 2) 调酒动作映射重做（确保振幅/频率/相位全可调）
- 文件：
  - `src/systems/mixology/mix-actions.ts`
  - `src/systems/mixology/mixology-system.ts`
  - `src/game/game-loop.ts`
- 变更：
  - 保留你确认的方案：`搅拌`改为相位工具（每次固定调整 `phaseOffset`）。
  - 明确动作职责：
    - 基酒：主控基础波形与振幅基底
    - 糖浆/柠檬：主控频率与对称性
    - 苏打：主控噪声
    - 冰块：主控衰减
    - 搅拌：主控相位
  - 将 action→ingredient/effect 映射收敛到 `mixology-system` 单点，`game-loop` 只负责分发，不重复硬编码。
- 原因：消除映射漂移，确保玩家操作与波形反馈是一致的一阶映射。

### 3) 波形生成升级（相位与复杂度可见）
- 文件：
  - `src/systems/wave/wave-generator.ts`
- 变更：
  - 在基础波形计算中引入 `phase`：`phaseTerm = basePhase + phaseOffset`。
  - 加入可控谐波叠加（低阶 2~3 次谐波），由 `harmonics` 控制占比。
  - 保持噪声为确定性伪随机（避免抖动），但噪声分布与谐波共同作用，形成后期复杂纹理。
- 原因：让后期“复杂波形”既可生成也可被操作逼近。

### 4) 拟合评分重构（解决“看起来像但分低”）
- 文件：
  - `src/systems/wave/wave-match.ts`
- 变更：
  - 新增“相位容忍”评分：在小范围位移窗口内取最佳对齐（best-shift MSE）。
  - 最终评分改为分项加权：
    - 形状匹配（含相位容忍）
    - 振幅误差
    - 频率误差
    - 相位误差
    - 噪声/衰减误差
  - 难度联动：高难订单逐步降低相位容忍窗口和容差阈值。
- 原因：评分逻辑与玩家视觉判断更一致，同时保留高难挑战性。

### 5) 复杂度分层与自动解锁（按天数阈值）
- 文件：
  - `src/systems/npc/npc-system.ts`
  - `src/content/orders.ts`
  - `src/game/game-state.ts`
  - `src/game/game-loop.ts`
- 变更：
  - 订单复杂度分层：
    - Day 1-3：单峰低噪、低相位偏移
    - Day 4-7：加入中等相位偏移与轻谐波
    - Day 8+：高谐波、相位偏移、噪声组合目标
  - 自动解锁策略（按天数阈值）：
    - Day 4：解锁第 4 种基酒
    - Day 7：解锁 2 个新道具（如苦精、摇壶）
    - Day 10：解锁高阶工具（如喷枪/点燃）
  - 在 `GameState` 增加可用道具池（或利用 `inventory` 显式标记可用项）。
- 原因：保证“后期目标更复杂”与“玩家可用工具更强”同步增长。

### 6) 新道具像素素材与调酒桌排布
- 文件：
  - `src/ui/pixel-assets.ts`
  - `src/ui/bar-renderer.ts`
- 变更：
  - 新增像素素材：新增基酒瓶型、苦精瓶、摇壶、喷枪等。
  - 调酒桌布局调整：
    - 左侧：基酒区（随解锁扩展）
    - 中部：酒杯/冰碗/核心操作区
    - 右侧：添加剂与高级工具区（随解锁显示）
  - 未解锁道具显示为“灰化锁定态”，避免 UI 跳变。
- 原因：满足“新增道具需有图像资源并合理排布”的明确要求。

### 7) UI 与反馈联动
- 文件：
  - `src/ui/hud-renderer.ts`
  - `src/ui/bar-renderer.ts`
- 变更：
  - 在混酒视图增加“当前波形参数微调提示”（振幅/频率/相位当前偏差方向）。
  - 结果面板增加分项反馈（形状、相位、频率等子分）。
- 原因：让玩家明确知道“该调什么”，减少盲调挫败感。

## Assumptions & Decisions
- 已锁定决策：
  - 相位控制采用“搅拌改为相位工具”。
  - 解锁节奏采用“按天数阈值自动解锁”。
  - 新增道具必须补像素资源并在调酒桌上合理排布。
- 约束：
  - 保持原生 TS + Canvas 架构，不引入重型框架。
  - 保持现有资源与存档系统兼容，新增字段通过版本迁移兜底。

## Verification Steps
- 玩法验证：
  1. 进入同一订单，分别只调整振幅/频率/相位，确认波形对应维度有明显、可预期变化。
  2. 人眼明显接近目标时，评分应显著高于改造前版本（对照 20 组样本单）。
  3. Day 1-3 基本可完成，Day 8+ 出现明显复杂波形挑战。
- 解锁验证：
  1. Day 4/7/10 自动解锁内容准确出现。
  2. 未解锁道具不可交互且视觉上有锁定态。
- 工程验证：
  - `npm run typecheck`
  - `npm run build`
  - 关键 UI 场景回归：`mixing_view`、`result`、`resource_settlement`。
